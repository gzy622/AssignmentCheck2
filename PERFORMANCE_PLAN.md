# 性能优化执行计划

## 目标

- 保持纯前端运行，不改数据结构、不改备份格式、不改版本机制。
- 优先处理高频同步开销，避免改动主网格渲染这种已经具备增量优化的路径。
- 每完成一个阶段就跑一次测试，保证改动可回滚、可验证。

## 顺序

### Phase 1: 草稿持久化批处理

- 目标：把 `saveRecoveryDraft()` 从“每次立即写 localStorage”改成“排队写入 + 退出前 flush”。
- 改动范围：`app.js`、必要时补 `tests/state.test.js`。
- 约束：保存的 JSON 结构保持不变，只调整写入时机。
- 验证：多次记录更新不会触发重复草稿写入；`beforeunload` 和显式 flush 仍能落盘。

### Phase 2: 名单编辑局部更新

- 目标：把名单弹窗里的整表刷新改成“单行更新 + summary 更新”。
- 改动范围：`actions.js`，必要时补 `tests/state.test.js`。
- 约束：保存输出格式保持不变，排序、删除、新增、清理空行行为不变。
- 验证：输入单个字段时只更新当前行和计数，不重建整表；结构性变化仍能完整刷新。

### Phase 3: 缓存失效收紧

- 目标：减少与统计无关的修改对统计缓存的清空。
- 改动范围：`app.js`。
- 约束：影响统计结果的修改仍必须失效缓存。
- 验证：改名称不应强制重算统计；改名单或记录仍应重算。

### Phase 4: 微优化

- 目标：只处理确认收益的小点，例如弹窗布局比较方式。
- 改动范围：`modal.js`，仅在前面阶段稳定后再做。
- 约束：不影响交互行为。
- 验证：现有测试全部通过即可。

## 执行规则

- 一次只推进一个 Phase。
- 先改代码，再补测试，再跑测试。
- 如果某个优化需要大范围联动，先停下来，不扩散到其他模块。
- 若测试失败，优先修复当前 Phase，不继续下一阶段。
