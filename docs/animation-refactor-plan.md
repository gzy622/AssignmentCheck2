# 全屏模态动画重构方案 V2

## 背景

基于安卓端实测，上一版方案虽然尝试统一平台表现，但核心思路仍然是“页面打开时同步做内容动画”。  
这类方案即使把位移、透明度和时长压低，依然会在安卓端遇到同一个根问题：

- 容器在动
- 内容在挂载
- 列表、图表、输入框还在首帧参与布局和绘制

结果就是动画参数变了，但首屏体感并不会真正接近 `ScorePad`。

`ScorePad` 之所以丝滑，不是因为它“动画更花”，而是因为它的设计结构更轻：

- 只有一个主容器做位移过渡
- 动画主要靠 `transform`
- 内部结构稳定，不在进入动画时批量创建复杂内容
- 交互层和内容层职责清楚

这次方案要完全放弃旧思路，改成和 `ScorePad` 同类的设计模型。

## 目标

- 全屏模态框在 Android、iOS、桌面浏览器上使用同一套进入和退出模型。
- 过渡体感向 `ScorePad` 靠拢，以“单容器滑入”为核心，而不是“整页多层淡入”。
- 允许页面内容分步骤加载，避免在动画首帧同时做重布局和重绘制。
- 保留“显示动画”开关，关闭时仍能直接显示完整内容。
- 不修改 `ScorePad` 本身。

## 新设计原则

### 1. 壳体先行，内容后到

全屏模态打开时，先出现稳定的模态壳体，只让壳体做一次位移动画。  
正文内容不再要求与壳体同时完整进场，而是分阶段挂载。

### 2. 只让一个重元素运动

参考 `ScorePad`，全屏模态进入阶段只允许一个主层做 `transform` 过渡：

- 全屏面板根节点

其他层只做以下两种处理之一：

- 不动
- 极轻的透明度切换，或者完全不做动画

### 3. 动画与渲染解耦

旧方案的问题不是单纯“动画不好看”，而是“动画时机和渲染时机重叠”。  
新方案把它们拆开：

1. 先打开壳体
2. 再挂轻内容
3. 最后按块补重内容

### 4. 安卓优先

设计参数先以安卓中端机可接受为基线，再向其他平台统一，不再做“桌面版精致、安卓版降级”的分叉方案。

## 核心方案

## 一、全屏模态改成“全高滑板”模型

不再使用当前 `dialog.full` 这种“背景层 + 卡片层 + 内部导航层 + 内容层”同时做动画的方式。  
改成接近 `ScorePad` 的全高滑板模型：

- 模态根层固定在视口内
- 内容板从屏幕底部整体滑入
- 板体高度直接占满可用视口
- 关闭时整板整体下滑离场

视觉上它像一个“全屏版 bottom sheet”，而不是“被展开的卡片页面”。

### 进入效果

- 遮罩层：
  - 要么不动画，直接到位
  - 要么只做极短的透明度过渡
- 主面板：
  - 初始位置在屏幕底部下方
  - 通过 `translate3d(0, 100%, 0)` 或带少量偏移的等价写法上滑到 `0`
  - 缓动沿用 `ScorePad / BottomSheet` 这种顺滑、偏手势感的曲线

### 退出效果

- 主面板整体下滑退出
- 遮罩同步收掉
- 不做正文子元素离场动画

### 预期优势

- 进入期只有一个大层移动
- 合成路径清楚
- 视觉重心稳定
- 安卓端更接近原生手势页或抽屉页的流畅感

## 二、内容加载改成三阶段

这是这次方案的重点，不是可选项。

### 阶段 A：壳体阶段

`Modal.show({ type: 'full' })` 被调用后，先只渲染最轻的结构：

- 面板容器
- 顶部导航壳
- 一个空白正文容器
- 必要的关闭按钮

这一阶段不挂载重列表、不画复杂图表、不批量插入表单块。

目标：

- 让动画起步时 DOM 足够轻
- 避免“动画刚开始就被内容布局拖慢”

### 阶段 B：首屏内容阶段

当面板进入动画完成，或者至少完成到主要可视位置后，再加载首屏轻内容：

- 标题
- 顶部工具区
- 首屏骨架或占位块
- 少量立即可交互控件

这一步的重点不是“全部内容 ready”，而是“用户已经感知到页面稳定打开”。

### 阶段 C：正文补全阶段

最后再分块加载真正重的内容：

- 长列表
- 图表
- 大量卡片
- 批量输入项

加载策略：

- 使用 `requestAnimationFrame` 分帧插入
- 必要时配合 `setTimeout(0)` 或小延迟分批执行
- 每批控制节点数，避免单帧超过预算
- 大列表继续沿用并强化现有的延迟渲染思想

## 三、首屏不是“空白”，而是“骨架到真内容”

如果阶段 A 和阶段 B 之间完全空白，用户会觉得像掉帧或加载慢。  
所以要引入很轻的壳内占位结构：

- 顶栏立即可见
- 正文先显示骨架块或静态占位片
- 真内容加载后替换占位

注意：

- 骨架本身不能复杂
- 不做波浪 shimmer
- 只做静态浅色块，避免再引入额外动画负担

## 四、把“全屏模态内容”改为可渐进挂载

当前很多全屏页面都是直接把完整 root 传给 `Modal.show()`。  
新方案建议改成“壳体 + 内容工厂”的调用模型。

### 建议的新接口方向

保留现有 `Modal.show()` 主入口，但给全屏模态增加渐进挂载能力，例如：

- `shellFactory`
- `contentFactory`
- `mountStrategy: 'immediate' | 'progressive'`
- `afterOpen`
- `onHydrate`

目标不是强行把业务接口做复杂，而是让 `Modal` 自己知道：

- 什么时候先开壳
- 什么时候挂轻内容
- 什么时候启动分块补全

## 五、把现有 `deferFullscreenWork` 升级为“分阶段调度器”

现在 `Actions.deferFullscreenWork()` 只是简单地延后任务开始。  
这还不够，因为它只解决“晚一点开始”，没有解决“分几步做完”。

建议升级成可表达阶段的调度模式，例如：

- `openShell`
- `mountAboveTheFold`
- `hydrateHeavyContent`

职责变化：

- 旧逻辑：延迟一次执行
- 新逻辑：按阶段执行多次，每次工作量受控

这样“作业项目管理”“学生名单”“小测趋势”都能用同一套节奏。

## 六、动画语言向 `ScorePad` 靠拢，但不照抄

参考点是它的机制，不是完全复制视觉。

### 要继承的部分

- 单主层位移
- `transform` 优先
- 打开关闭路径短
- 拖拽式、手势感的缓动风格

### 不直接照抄的部分

- 全屏模态不是半高面板，不能保留明显抽屉留边
- 顶栏和正文布局不同，需要独立的壳体结构
- 某些页面仍需要自己的内部延迟渲染，而不是一口气塞满

## 七、明确放弃的旧做法

以下设计在本轮应明确废弃：

- 全屏模态内部导航、正文块逐项错峰进入
- 全屏模态卡片缩放进入
- Android 单独走 `lite-transitions`
- 打开瞬间直接插入完整重内容树
- 用“更短更轻的 CSS 动画”来掩盖渲染卡顿

这些做法都不符合这次“像 `ScorePad` 一样顺”的目标。

## 组件层面的新规划

## 1. `Modal`

需要升级为“壳体驱动”的全屏模态控制器：

- 增加全屏模态专用壳体节点
- 支持壳体先打开、内容后挂载
- 关闭时能取消未完成的补全任务
- JS 关闭等待时长改为与新滑动时长一致

## 2. `ActionViews`

全屏页面视图构造函数要从“直接返回完整页面”转成“可分阶段输出页面”：

- 立即阶段：返回壳体和骨架
- 延迟阶段：返回重内容挂载函数

## 3. `Actions`

页面打开逻辑要能识别不同内容重量：

- 轻页面可以一次挂载
- 重页面必须走渐进挂载

尤其是这些页面要优先接入：

- 作业项目管理
- 编辑学生名单
- 小测趋势

## 4. `styles.css`

动画只保留最小集合：

- 全屏板进入
- 全屏板退出
- 极轻遮罩过渡
- 骨架到内容的轻量显现

不再保留全屏模态内部多层关键帧矩阵。

## 实施步骤

1. 先定义新全屏模态的运动模型，统一为“单主板滑入滑出”。
2. 重写 `modal.js` 的全屏模态生命周期，引入壳体阶段和内容挂载阶段。
3. 把 `Actions.deferFullscreenWork()` 升级成分阶段调度。
4. 先让“作业项目管理”接入新模式，验证安卓端体感。
5. 再迁移“编辑学生名单”“小测趋势”等重页面。
6. 最后清理旧的 `dialog.full` 多层动画、`lite-transitions` 和错峰关键帧。

## 验收标准

- 安卓端打开全屏模态时，主观体感接近 `ScorePad` 的顺滑度。
- 动画开始时不再伴随明显首帧卡顿。
- 页面打开后用户能先看到稳定外壳，再看到首屏内容，再看到重内容补齐。
- 平台间不再有不同动画逻辑分支。
- 动画关闭时，全屏页面直接显示完整内容，不进入渐进动画流程。

## 测试重点

正式实施时需要重点验证：

- Android Chrome 真机打开“作业项目管理”
- Android Chrome 真机打开“编辑学生名单”
- Android Chrome 真机打开“小测趋势”
- 快速连续打开关闭时，未完成挂载任务是否被正确取消
- 动画关闭开关打开/关闭时，内容是否仍能完整显示

## 结论

这次新的设计方案不再围绕“怎样把旧的淡入上浮动画调顺”，而是直接换模型：  
把全屏模态改成类似 `ScorePad` 的全高滑板，用单主层滑入承接全部过渡，再让页面内容在动画后按阶段挂载。  
真正解决安卓卡顿，关键不是继续修饰 CSS，而是让动画和重内容渲染彻底分家。
