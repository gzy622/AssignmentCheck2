### 2.2 关键依赖分析

**强依赖（必须存在）：**
- `State` → `LS` (数据持久化)
- `Modal` → `ActionViews` (UI构建)
- `ScorePad` → `State` (数据更新)

**循环依赖风险：**
```
State → Modal (通过 view 回调)
Modal → BackHandler (关闭处理)
BackHandler → Modal (检查状态)
```

## 三、代码冗余与性能瓶颈

### 3.1 发现的冗余代码

#### 问题 1：重复的 ID 生成逻辑
```javascript
// app.js:220
let id = Date.now();
while (this.asgMap.has(id)) id++;

// 多处使用 Date.now() 生成 ID，无统一封装
```

**改进建议：** 创建 `IdGenerator` 工具类

#### 问题 2：重复的数据验证
```javascript
// app.js 多处出现类似的验证逻辑
if (!draft || typeof draft !== 'object' || !Array.isArray(draft.list)...)
// 在 getRecoveryDraft, normalizeAsg 等函数中重复
```

**改进建议：** 提取 `Validator` 模块

#### 问题 3：缓存机制过度复杂
```javascript
// app.js:14-22
_metricsToken: 0,
_metricsCache: new Map(),
_scoreRangeCache: new Map(),
// ... 还有多个 dirty 标记
```

**分析：** 缓存系统过于复杂，实际收益可能不明显

### 3.2 性能瓶颈识别

| 位置 | 问题 | 影响 | 优先级 |
|------|------|------|--------|
| `app.js:parseRoster()` | O(n) 遍历，大数据量时卡顿 | 名单加载 | 高 |
| `actions.js:renderList()` | DOM 批量操作无虚拟列表 | 任务多时卡顿 | 高 |
| `modal.js:scheduleLayout()` | 频繁 resize 计算 | 低端设备卡顿 | 中 |
| `scorepad.js:_renderKeypad()` | 每次模式切换都重建 DOM | 切换延迟 | 低 |

### 3.3 内存泄漏风险

```javascript
// modal.js: ProgressiveController
// 虽然做了清理，但以下情况可能泄漏：
- 组件卸载时 cleanup 未完全执行
- 全局事件监听未移除（已处理较好）
- setTimeout 未清理边界情况
```

## 四、可优化的设计模式

### 4.1 当前架构问题

**问题：State 对象过于庞大（800+ 行）**
```javascript
const State = {
    // 数据状态
    list: [], roster: [], data: [], curId: null,
    // UI 状态
    mode: 'name', scoring: false, animations: true,
    // 缓存
    asgMap: new Map(), rosterIndexMap: new Map(),
    // 内部状态
    _persistTimer: 0, _draftTimer: 0, ...
    // 方法（80+ 个）
    init() {}, save() {}, parseRoster() {}, ...
}
```

### 4.2 推荐重构方案

```
当前模式：单一 State 对象
    ↓
建议模式：分层架构

┌─────────────────────────────────────┐
│         Presentation Layer          │
│    (UI 组件: Modal, ScorePad)       │
└─────────────────────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────┐
│          Business Layer             │
│  (Services: RosterService,          │
│           AssignmentService,        │
│           ExportService)            │
└─────────────────────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────┐
│           Data Layer                │
│  (Storage: LocalStorageAdapter,     │
│           RecoveryManager)          │
└─────────────────────────────────────┘
```

### 4.3 具体重构建议

#### 建议 1：拆分 State 为多个 Service

```javascript
// 建议结构
const RosterService = {
    list: [],
    roster: [],
    rosterIndexMap: new Map(),
    parseRoster() {},
    saveRoster() {},
};

const AssignmentService = {
    data: [],
    asgMap: new Map(),
    curId: null,
    addAsg() {},
    removeAsg() {},
    updateAsgMeta() {},
};

const PreferenceService = {
    prefs: {},
    animations: true,
    savePrefs() {},
    applyPrefs() {},
};
```

#### 建议 2：引入事件总线替代直接调用

```javascript
// 当前模式（紧耦合）
State.updRec(id, record);
UI.renderStudent(id); // 直接调用

// 建议模式（松耦合）
EventBus.emit('record:updated', { id, record });
// UI 层监听事件后更新
```

#### 建议 3：虚拟滚动优化大数据量

```javascript
// 当前：全部渲染
list.replaceChildren();
asgs.forEach(asg => list.appendChild(createCard(asg)));

// 建议：虚拟滚动
VirtualList.render({
    container: list,
    items: asgs,
    itemHeight: 80,
    renderItem: (asg) => createCard(asg)
});
```

## 七、总结

### 当前状态
- ✅ **功能完整性**：功能丰富，覆盖教学场景需求
- ✅ **代码整洁度**：命名规范，结构清晰
- ⚠️ **架构合理性**：State 过于庞大，需要拆分
- ⚠️ **性能优化**：大数据量场景需要虚拟列表

### 核心建议
1. **立即修复**：[app.js:47-48](../../app.js#L47-L48) 的空 catch 块
2. **优先优化**：实现虚拟列表解决大数据卡顿
3. **中期重构**：将 State 拆分为多个 Service
4. **长期规划**：考虑 TypeScript 迁移

这个项目从 MVP 到完整功能的演进路径清晰，代码质量整体良好，主要问题在于核心 State 对象随着功能增加变得臃肿，需要适时分层解耦。
